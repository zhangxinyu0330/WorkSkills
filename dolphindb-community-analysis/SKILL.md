# DolphinDB 社区聊天记录分析 Skill

## 目标

从一个或多个微信群聊天记录（.docx）中，提取有价值的技术问答，总结共性问题，并生成适合发布到英文技术平台的提问建议，最终输出统一格式的三段式 Markdown 报告。

---

## 工作流程（三步走）

### Step 1：读取所有 .docx 文件

使用 `pandoc` 将每个上传的 .docx 转为 markdown 文本：

```bash
pandoc "/mnt/user-data/uploads/文件名.docx" -t markdown
```

对每个文件依次执行，提取全部聊天内容。忽略图片标记（`![IMG_xxx](...)`）。

---

### Step 2：按文件逐群整理问答（各群问答表）

**分析规则：**
- 以单个问题为维度，一个问题可能跨多轮对话，合并写在同一行
- 识别提问人（提出疑问的人）与回答人（给出解释或方案的人），不能混淆
- 如对话未解决问题，回答栏注明"对话截止，问题未完全解决"
- 如有文档链接，提取到 ask/jira 链接列

**每个文件生成一张表，表头：**

| 提问时间 | 提问人 | 问题 | 回答 | 回答人 | ask/jira链接 |
|----------|--------|------|------|--------|--------------|

---

### Step 3：跨群汇总分析，生成两张总结表

#### 表 A：共性问题总结表（面向团队内部改进）

从所有群的问题中，提炼出有共性或高价值的问题，归纳为团队可行动的改进建议。

**表头：**

| 类别 | 共性问题（一句话提炼） | 典型影响 | 建议动作 |
|------|----------------------|----------|----------|

**分析维度建议：**
- 文档质量（文档缺失、描述不清）
- 功能边界不清晰（用户误用 API 或功能）
- 配置理解偏差（参数组合效果不符预期）
- 运维排查路径缺失（无自助排查文档）
- 产品体验问题（注册入口、恢复时长等）

**标注规则：** 若某问题已有对应 Jira/内部 ticket，在"共性问题"列用「🔵 已提 Jira」标注。

---

#### 表 B：英文平台提问建议表（面向 Stack Overflow / DBA）

针对适合公开提问的问题，生成英文标题，并判断适合发布的平台。

**表头：**

| 类别 | 具体问题（中文标题） | 具体问题（英文标题） | 微信群对应提问 | 适合平台 |
|------|---------------------|---------------------|---------------|----------|

**平台判断规则：**
- **Stack Overflow**：数据分析类问题 + 有代码/API 用法（如函数用法、写法差异、数据结构操作）
- **Database Administrators**：运维类问题 + 有报错信息（如节点宕机、启动失败、恢复时间异常）

---

## 最终输出结构（严格遵循）

输出为连续 Markdown，分为三个区块，顺序固定：

```
## DolphinDB 社区高价值问题总结（共性问题）

> 以下是根据整理的社区问题总结的有价值或具有共性的问题（标🔵的已提 Jira）：

[表 A：共性问题总结表]

---

## AI 建议提到 Stack Overflow 和 DBA 上的问题列表

[表 B：英文平台提问建议表]

---

## 各群聊天记录问答整理

### [文件名1（去掉.docx后缀）]

[该文件对应的问答表]

### [文件名2]

[该文件对应的问答表]

...（每个文件一个 H3 小节）
```

---

## 注意事项

- **不要遗漏任何文件**：即使某个文件内容极少（只有 1~2 条对话），也要生成对应的问答表
- **重复文件去重**：若上传了相同文件名的文件，只处理一次
- **图片内容忽略**：聊天中的图片截图无法读取，在问题/回答描述中可注明"（附截图）"
- **提问人与回答人准确性**：提问人是最先表达困惑或提出问题的一方，回答人是给出方案或解释的一方；如系统自动消息或闲聊，跳过不整理
- **中文输出为主**：各群问答表及共性表均用中文；仅表 B 的英文标题列使用英文
