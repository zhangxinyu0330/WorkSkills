# DolphinDB 插件/模块推文写作

根据开发者填写的插件/模块问卷，撰写一篇结构统一、语气一致的 DolphinDB Marketplace 生态宣传推文（微信公众号格式）。

## 输入来源

用户通常会提供以下文件，**每个文件均需先提取内容再写作**：

| 文件 | 用途 |
|------|------|
| 插件/模块信息问卷（`.docx`） | 必须，提供插件核心信息 |
| 开发者故事文件（`.docx`） | 可选，提供"关于开发者"板块素材 |

**提取命令（每个文件单独执行）**：
```bash
pandoc /mnt/user-data/uploads/文件名.docx -o /tmp/output.md && cat /tmp/output.md
```

## 信息提取清单

从输入材料中识别以下字段（对应问卷结构）：

| 字段 | 说明 |
|------|------|
| 插件/模块名称 | 上架名称，用于标题和正文 |
| 核心定位 | 一句话描述，提炼为开篇破题句 |
| 核心功能亮点 | 3 点以内，转化为"核心亮点"列表 |
| 解决的用户核心痛点 | 开篇"现实痛点"段落的原材料 |
| 适用用户/场景 | "谁适合用"板块 |
| 免费/付费说明 | 在"立即体验"板块末尾标注 |
| 关键使用优势 | 融入"核心亮点"或"快速上手"描述中 |
| Demo 示例（代码） | 直接放入"快速上手"代码块 |
| 使用文档 / 技术支持 | 填入"立即体验"板块 |
| 开发者信息（如有） | 可选，生成"关于开发者"板块 |

## 推文结构模板

严格按照以下 8 个板块顺序输出，不得随意增减或换序：

### 1. 标题
固定结尾：`[插件名] 正式上架 DolphinDB Marketplace！`
- 结尾前的内容自由发挥，可以是目标用户、使用场景、核心价值、悬念设问等任意角度
- 标题整体不超过 30 字，语气简洁有力

### 2. 开篇痛点（2–3 段）
目标是让目标用户读完第一屏就产生共鸣，句式不作强制要求，灵活处理。把握以下三个方向：
- **切入点**：从目标用户的真实处境出发，可以是一个场景、一个操作步骤、一句自问，或一个反差
- **痛点展开**：描述现实中的繁琐或障碍，要具体（有工具名、有步骤、有代价），不夸张
- **转折收尾**：自然引出插件的出现，语气上完成"问题 → 解法"的过渡即可，不必套用固定句式

### 3. 重磅上架（2–3 句）
固定结构：
```
**[插件名] 现已正式上线 DolphinDB Marketplace！**

[一句话介绍：由谁开发 / 封装了什么 / 解决什么核心问题。]

**[核心价值口号，15字以内，加粗]**
```

### 4. 核心亮点（列表）
- 3–5 条，每条格式：`- **亮点标题**：具体说明`
- 内容来自"核心功能亮点"+"关键使用优势"，去重合并
- 禁止使用"革命性""颠覆""游戏规则改变者"等夸张词

### 5. 快速上手（代码驱动）
固定结构：
1. 安装方式（`installPlugin` 或 `installModule` 命令）
2. 依赖要求（如有）
3. 完整使用链路代码块，带步骤编号注释
- 代码从问卷 Demo 示例字段获取，保持原样，仅添加注释
- 如问卷无代码，根据亮点描述推导，并在正文标注"示意代码，请以文档为准"

### 6. 谁适合用这个插件？
固定结构：
```
**目标用户**：[来自问卷"适用用户"字段]

**典型场景**：
- 场景一
- 场景二
- 场景三
```

### 7. 立即体验
固定结构：
```
- **项目地址**：[链接]
- **使用文档**：[链接或说明]
- **技术支持**：[邮箱或渠道]

> **定价说明**：[免费/付费/试用说明]
```

### 8. 共建生态（固定结尾，每篇一致）
此板块内容固定，每篇推文保持一致，用于强化 DolphinDB 生态品牌：

```markdown
## 共建生态：从使用者到贡献者

社区的繁荣源于每一份分享。我们诚挚邀请开发者、技术团队加入生态共建：

- **对于社区用户**：灵活调用各类免费插件/模块资源，适配日常业务需求，无需复杂开发即可享受高性能工具加持
- **对于企业用户**：精准补齐业务短板，聚焦核心业务逻辑创新，大幅提升开发效率与流程灵活性
- **对于开发者/技术团队**：这里是技术成果转化的优质舞台。免费分享可积累行业口碑与影响力，定制化插件/模块更可实现技术变现

**立即开启高效开发之旅**

- 开发者手册：[开发者手册 - DolphinDB](https://docs.dolphindb.cn)

点击**阅读原文**，访问 DolphinDB Marketplace 体验插件。

*扫码添加 DolphinDB 小助手，进入技术/高校交流群*
```

## 可选板块：关于开发者

**触发条件**：用户提供了独立的开发者故事文件（`.docx`）。

**插入位置**：在"谁适合用"之后、"立即体验"之前。

**素材提取优先级**（从开发者故事文件中按顺序找）：
1. **引言**：优先取"对犹豫加入的伙伴有什么建议"或"印象深刻的话"，语气真实接地气
2. **身份与背景**：职业、使用 DolphinDB 年限、核心业务场景
3. **开发动机**：从"什么契机促使你开发"提炼，突出从真实业务痛点出发
4. **技术亮点**：从"最大的挑战或成就"提炼，保留具体技术细节（如库冲突、API 设计等），有细节才有可信度
5. **后续计划**：从"未来是否继续迭代"提炼，1 句即可

**结构**：
```
## 关于开发者

> "[引言原话或精炼，保留开发者语气]" —— [姓名]，[身份]

[第一段：身份 + DolphinDB 使用年限 + 核心业务场景，2–3 句]

[第二段：开发动机 + 技术挑战细节，2–3 句]

[第三段：后续计划，1–2 句]
```

## 语气与风格规范（继承自 article-writing skill）

- 句子简短直接，段落不超过 4 行
- 用破折号（——）和省略号制造节奏感
- 痛点描述要贴近真实场景，不用"痛苦""煎熬"等情绪词
- 亮点描述要有技术细节，不写空洞的"高效""稳定"
- **禁止使用**：革命性、颠覆性、游戏规则改变者、尖端、强大的、功能丰富的

## 品牌安全规范（必须遵守）

痛点描述的矛头永远指向**行业现状、第三方工具局限、用户自身面临的环境问题**，不得暗示或明示 DolphinDB 本身存在缺陷。具体要求：

**开篇痛点段落**：
- ✅ 写：官方 SDK / 第三方数据源 / 行业工具 的平台限制或流程繁琐
- ✅ 写：用户自己的工作环境带来的挑战（如"Windows 本地开发没有原生方案"）
- ❌ 不写：DolphinDB 缺少某功能、DolphinDB 不支持某平台、DolphinDB 官方插件的不足

**开发者故事板块**：
- 开发者问卷中可能出现"希望 DolphinDB 能改进 XX"等反馈，**一律不写入推文**
- 若开发动机涉及"官方插件不支持 Windows"，改写为"Windows 本地开发缺少原生接入方案"，主语从 DolphinDB 改为用户处境
- 技术挑战部分可以保留具体技术细节（如 ABI 冲突），但不得将挑战归咎于 DolphinDB 设计

## 输出格式

- 输出为 Markdown 格式
- 使用 `##` 作为各板块标题级别
- 代码块使用三反引号包裹
- 最终保存为 `/mnt/user-data/outputs/[插件名]推文.md`

## 质量检查（输出前必查）

- [ ] 标题是否包含具体目标用户群体？
- [ ] 开篇是否有真实的痛点场景描述？
- [ ] 代码块是否来自问卷原始 Demo？
- [ ] "共建生态"板块是否与模板完全一致？
- [ ] 是否存在夸张宣传词或编造内容？
- [ ] 定价信息是否与问卷一致？
- [ ] 全文是否存在暗示 DolphinDB 本身存在缺陷的表述？（若有，必须改写）
- [ ] 开发者故事中的"改进建议"类内容是否已全部过滤？
