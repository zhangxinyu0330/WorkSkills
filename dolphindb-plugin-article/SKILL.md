# DolphinDB 插件/模块推文写作

根据开发者填写的插件/模块问卷，撰写一篇结构统一、语气一致的 DolphinDB Marketplace 生态宣传推文（微信公众号格式）。

## 输入来源

用户通常会提供以下一种或多种输入：
- `.docx` 问卷文件（标准信息收集模板）
- PDF 或文本形式的插件介绍材料
- 直接粘贴的插件信息

**如果输入是 `.docx` 文件**，先用以下命令提取内容，再继续写作：
```bash
pandoc /mnt/user-data/uploads/文件名.docx -o /tmp/plugin_info.md && cat /tmp/plugin_info.md
```

## 信息提取清单

从输入材料中识别以下字段（对应问卷结构）：

| 字段 | 说明 |
|------|------|
| 插件/模块名称 | 上架名称，用于标题和正文 |
| 核心定位 | 一句话描述，提炼为开篇破题句 |
| 核心功能亮点 | 3 点以内，转化为"核心亮点"列表 |
| 解决的用户核心痛点 | 开篇"现实痛点"段落的原材料 |
| 适用用户/场景 | "谁适合用"板块 |
| 免费/付费说明 | 在"立即体验"板块末尾标注 |
| 关键使用优势 | 融入"核心亮点"或"快速上手"描述中 |
| Demo 示例（代码） | 直接放入"快速上手"代码块 |
| 使用文档 / 技术支持 | 填入"立即体验"板块 |
| 开发者信息（如有） | 可选，生成"关于开发者"板块 |

## 推文结构模板

严格按照以下 8 个板块顺序输出，不得随意增减或换序：

### 1. 标题
格式：`[目标用户]，终于等来了这个——[插件名] 正式上架 DolphinDB Marketplace！`
- 目标用户从"适用用户"字段提炼，要具体（如"Windows 量化人"、"iFinD 用户"）
- 标题不超过 30 字

### 2. 开篇痛点（2–3 段）
- **第一段**：一句话点明"这件事本不该这么难"，切入目标用户的核心场景
- **第二段**：具体描述现实中的繁琐流程或痛点，用"→"串联多个步骤，语气贴近真实场景，不夸张
- **第三段**：转折破题，固定句式：`现在，这套流程可以直接[简化为一步 / 走通]——`

### 3. 重磅上架（2–3 句）
固定结构：
```
**[插件名] 现已正式上线 DolphinDB Marketplace！**

[一句话介绍：由谁开发 / 封装了什么 / 解决什么核心问题。]

**[核心价值口号，15字以内，加粗]**
```

### 4. 核心亮点（列表）
- 3–5 条，每条格式：`- **亮点标题**：具体说明`
- 内容来自"核心功能亮点"+"关键使用优势"，去重合并
- 禁止使用"革命性""颠覆""游戏规则改变者"等夸张词

### 5. 快速上手（代码驱动）
固定结构：
1. 安装方式（`installPlugin` 或 `installModule` 命令）
2. 依赖要求（如有）
3. 完整使用链路代码块，带步骤编号注释
- 代码从问卷 Demo 示例字段获取，保持原样，仅添加注释
- 如问卷无代码，根据亮点描述推导，并在正文标注"示意代码，请以文档为准"

### 6. 谁适合用这个插件？
固定结构：
```
**目标用户**：[来自问卷"适用用户"字段]

**典型场景**：
- 场景一
- 场景二
- 场景三
```

### 7. 立即体验
固定结构：
```
- **项目地址**：[链接]
- **使用文档**：[链接或说明]
- **技术支持**：[邮箱或渠道]

> **定价说明**：[免费/付费/试用说明]
```

### 8. 共建生态（固定结尾，每篇一致）
此板块内容固定，每篇推文保持一致，用于强化 DolphinDB 生态品牌：

```markdown
## 共建生态：从使用者到贡献者

社区的繁荣源于每一份分享。我们诚挚邀请开发者、技术团队加入生态共建：

- **对于社区用户**：灵活调用各类免费插件/模块资源，适配日常业务需求，无需复杂开发即可享受高性能工具加持
- **对于企业用户**：精准补齐业务短板，聚焦核心业务逻辑创新，大幅提升开发效率与流程灵活性
- **对于开发者/技术团队**：这里是技术成果转化的优质舞台。免费分享可积累行业口碑与影响力，定制化插件/模块更可实现技术变现

**立即开启高效开发之旅**

- 开发者手册：[开发者手册 - DolphinDB](https://docs.dolphindb.cn)

点击**阅读原文**，访问 DolphinDB Marketplace 体验插件。

*扫码添加 DolphinDB 小助手，进入技术/高校交流群*
```

## 可选板块：关于开发者

**触发条件**：问卷中提供了开发者姓名、背景描述或引言。

插入位置：在"立即体验"之前，"谁适合用"之后。

结构：
```
## 关于开发者

> "[开发者引言，保留原话或精炼]" —— [姓名]，[职位/机构]

[2–3 句背景介绍：工作年限、专业领域、开发此插件的动机]

[1 句后续计划，如有]
```

---

## 语气与风格规范（继承自 article-writing skill）

- 句子简短直接，段落不超过 4 行
- 用破折号（——）和省略号制造节奏感
- 痛点描述要贴近真实场景，不用"痛苦""煎熬"等情绪词
- 亮点描述要有技术细节，不写空洞的"高效""稳定"
- **禁止使用**：革命性、颠覆性、游戏规则改变者、尖端、强大的、功能丰富的

## 输出格式

- 输出为 Markdown 格式
- 使用 `##` 作为各板块标题级别
- 代码块使用三反引号包裹
- 最终保存为 `/mnt/user-data/outputs/[插件名]推文.md`

## 质量检查（输出前必查）

- [ ] 标题是否包含具体目标用户群体？
- [ ] 开篇是否有真实的痛点场景描述？
- [ ] 代码块是否来自问卷原始 Demo？
- [ ] "共建生态"板块是否与模板完全一致？
- [ ] 是否存在夸张宣传词或编造内容？
- [ ] 定价信息是否与问卷一致？
